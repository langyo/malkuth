# Единая архитектура супервизии, скользящего обновления и репликации

> **Область.** Это *платформенного уровня* дизайн: он охватывает `core`
> (entelecheia / scepter), `webui` (shittim-chest / chest) и `router`
> (evernight). Архитектурные документы отдельных проектов находятся в их
> собственных подкатегориях `core/`, `webui/`, `router/`; этот документ
> определяет общий слой жизненного цикла / супервизии, который все они
> используют.

## 1. Предпосылки и цели

Все три проекта структурно однородны — это **Rust (edition 2024, MSRV
1.85) + axum 0.8 + tokio + JSON-RPC через Unix-сокеты / WebSocket**, и
все уже используют крейт `arona` в качестве протокольного слоя. Именно
эта однородность делает *единую* механизм супервизии, который стоит
сделать один раз и переиспользовать в трёх местах.

Этот механизм должен обслуживать четыре пересекающиеся потребности,
выраженные как одно целостное средство:

1. **Балансировка нагрузки** — запуск нескольких идентичных экземпляров
   одной и той же программы для совместного выполнения работы, координация
   через IPC при общем доступе к базе данных / конфигурации / runtime-состоянию.
2. **Скоординированные записи** — когда один экземпляр собирается записать
   общий файл, он должен уведомить остальные и взять замок, чтобы
   параллельные изменения не повредили состояние.
3. **Скользящее обновление** — при выходе нового официального релиза (или
   свежескомпилированного локального отладочного сервера) новый и старый
   бинарники могут сосуществовать; старый процесс завершает текущую работу,
   затем выходит и передаёт рабочий набор новому процессу.
4. **Периферийная отказоустойчивость** — на устройстве (в частности, на
   шлюзах evernight) два процесса могут работать как лидер/ведомый, чтобы
   падение одного не выводило из строя всё устройство.

### 1.1 Текущее состояние (пробел, который закрывает данный дизайн)

Аудит кода выявил одни и те же три недостатка во **всех трёх** проектах:

| Возможность | entelecheia (scepter) | shittim-chest (chest) | evernight |
|---|---|---|---|
| Обработка сигналов | только `ctrl_c` (`shutdown.rs:17`) | только `ctrl_c` (`api.rs:465`) | только `ctrl_c` (`api/mod.rs:109`) |
| Логика дрейна | дрейнит HTTP, но не WS / фоновые задачи | то же | нет |
| Передача listener fd | нет | нет | нет |
| `/readyz` с битом дрейна | нет | есть `/api/health`, без бита дрейна | нет |

Главная проблема: поскольку перехватывается только `SIGINT`, `docker stop` /
`systemctl restart`, которые отправляют **`SIGTERM`**, полностью обходят
мягкое завершение и жёстко убивают процесс после grace-периода. Исправление
только этого — изменение с наивысшим эффектом.

### 1.2 Существующие активы для переиспользования

- `entelecheia/packages/cli/src/evernight_daemon.rs` — на сегодняшний день
  наиболее полный чертёж самоперезапуска: PID-файл блокировки +
  self-reexec + ожидание готовности + fallback `SIGTERM`→`SIGKILL`.
- `entelecheia/packages/scepter/src/daemon/health_daemon.rs` — **очередь
  JSON-manifest-файлов**, выполняющая скользящее обновление контейнеров
  (языко-независимый примитив обновления).
- `entelecheia/packages/shared/infra_jsonrpc` — транспортный слой JSON-RPC
  через Unix-сокеты.
- `shittim-chest/packages/core/src/proxy/upstream_pool.rs` — реестр
  нескольких эндпоинтов с повторным подключением через экспоненциальный
  backoff; готовый шаблон балансирующего клиента.
- `evernight/src/model_server.rs` — конечный автомат жизненного цикла
  ресурса `Running/Starting/Stopped/Failed` с развёртыванием → ожиданием
  здоровья → остановкой старого; шаблон скользящего обновления на уровне
  приложения в репозитории.

## 2. Теоретическая основа

Этот механизм — не одна теория, а композиция хорошо зарекомендовавших
себя индустриальных паттернов, у каждого из которых есть эталонная
реализация. Называя их явно, мы позволяем дизайну унаследовать их
проверенную семантику:

| Потребность | Индустриальный термин | Эталонная реализация |
|---|---|---|
| Новый и старый процессы сосуществуют; старый завершает текущее и выходит | **мягкое завершение / дрейн (drain) + скользящее обновление** | Kubernetes Deployment, nginx / unicorn |
| «Красно-синий маркер» (название, всплывшее в обсуждении) | **blue-green deployment** (две параллельные среды, переключение указателя трафика). Описанное поведение прогрессивного угасания ближе к **скользящему обновлению + дрейну**, чем к blue-green. | |
| Новый процесс принимает на себя тот же порт без разрыва соединений | **socket activation / наследование fd / `SO_REUSEPORT`** | systemd, nginx `USR2`, envoy hot restart |
| «Уведомить другого и заблокировать перед записью» | **advisory lock (`flock`/`fcntl`) / блокировка строки БД / lease** | POSIX advisory locks, `pg_advisory_lock` |
| Самовосстановление процесса | **дерево супервизии (OTP) / systemd / kubelet** | Erlang/OTP, systemd, s6, immortal |
| leader/follower, чтобы крах не убивал сервис | **выборы лидера на основе lease + fencing** | Chubby lease, выборы лидера Raft (подмножество), keepalived/VRRP, Pacemaker |
| Ресурс удерживается одним процессом, перезапускается при крахе | **«let it crash» + перезапуск супервизором** | деревья супервизии Erlang/OTP, systemd `Restart=always` |

**Индустриальный стандартный рецепт скользящего обновления** (стоит
скопировать дословно) — это рецепт Kubernetes:
`maxSurge + maxUnavailable + readinessProbe + preStop hook + SIGTERM
мягкое завершение + grace-период + PodDisruptionBudget`. То, что мы
строим, — это self-hosted, однохостовый / для небольшого кластера
вариант этого рецепта.

**Горячее обновление nginx** — учебник по теме «новый и старый
сосуществуют, затем старый дрейнит»: `USR2` запускает новый master,
который наследует listen fd → `WINCH` мягко останавливает старые
воркеры → `QUIT` выводит старый master из эксплуатации. Поток в §7
структурно идентичен.

## 3. Общая архитектура

Дизайн сходится к единой элегантной основе: **дерево супервизии
повсюду; единственное отличие — является ли супервизор равноправной
репликой или парой leader/follower.**

```
                    ┌─────────────────────────────────────────┐
   Общая база       │ семантика сигналов · дрейн              │  ← идентично для
   (Layer 1 / 3)    │ /healthz · /readyz (с битом дрейна)     │     всех трёх проектов
                    │ socket activation + 3 адаптера развёртывания │
                    └─────────────────────────────────────────┘
                                   │
          ┌────────────────────────┴────────────────────────┐
          ▼                                                  ▼
   ┌────────────────┐                              ┌──────────────────┐
   │ Подсистема A   │  супервизоры РАВНОПРАВНЫ      │ Подсистема B     │  супервизор —
   │ Replica        │  (active-active)              │ Leader/Follower  │  leader или follower
   │                │  ← серверная LB +             │                  │  ← периферийная
   │                │    скользящее обновление      │                  │    отказоустойчивость
   └───────┬────────┘                              └─────────┬────────┘
           │                                                  │
           └──────────────────┬───────────────────────────────┘
                              ▼
               ┌──────────────────────────────┐
               │ Унифицированная абстракция   │  ← каждый «ресурс-дочерний процесс»
               │ Worker                       │     во всех трёх проектах
               │ FSM жизненного цикла +       │     cosmos / pglite-proxy / воркеры
               │ супервизируемый перезапуск   │     отдельных протоколов
               │ (permanent/transient) +      │
               │ rate-limit скользящим окном  │
               └──────────────────────────────┘
```

### 3.1 Ключевое упрощение: скользящее обновление — *не* третья система

Как только дерево супервизии становится основой, потребности пользователя
сводятся к **двум** подсистемам, а не к трём:

- **Подсистема A — Replica.** На серверной стороне работает N идентичных
  равноправных экземпляров, разделяющих нагрузку. *Балансировка нагрузки
  и скользящее обновление — одна и та же подсистема:* скользящее обновление
  — это просто «число реплик временно +1 → дрейним одну старую реплику →
  повторяем». Это операция `maxSurge/maxUnavailable` из Kubernetes,
  выраженная как добавление/удаление реплик.
- **Подсистема B — Leader/Follower.** На периферии (устройство evernight)
  работают два процесса, один лидер и один ведомый, для отказоустойчивости.
  Лидер эксклюзивно владеет физическим вводом-выводом; ведомый ожидает.

Отдельной «системы скользящего обновления» нет: это операция обслуживания
подсистемы A (и, с иной семантикой, подсистемы B через переключение лидера).

### 3.2 Многоуровневое представление

| Уровень | Подсистема A (Replica) | Подсистема B (Leader/Follower) | Общая? |
|---|---|---|---|
| **L1** жизненный цикл (сигналы / дрейн / пробы) | то же | то же | **общая** |
| **L3** бесшовная передача (socket activation) | каждая реплика получает fd от systemd | leader→follower захват fd (продвинутое) | **общая** |
| **L2** координация | **2a** реестр равноправных узлов + общий замок (`pg_advisory`) | **2b** выборы по lease + эксклюзивный ресурс + реестр leader/follower | **разделяется** (один trait, разная политика) |
| **L4** оркестрация | **4a** масштабирование реплик / скользящее обновление | **4b** переключение при отказе | **разделяется** |

Вывод: **L1 и L3 полностью общие; L2/L4 разделяются.** `CoordinationLock`
— один и тот же trait в 2a и 2b: в A координирует параллельные записи, в
B используется как lease лидера. Это объединение trait и есть точка, в
которой «общие принципы» материализуются.

## 4. Принадлежность крейта

Цель пользователя «положить это в arona» требует разделения, поскольку
**arona сегодня — это чисто протокольный/типовой крейт**: только
зависимости `serde` / `ts-rs` / `schemars`, `lib.rs:5` требует «каждый тип
определяется в entelecheia и потребляется shittim-chest», и для публикации
на crates.io он `exclude`-ит все непротокольные артефакты. Внедрение
runtime-логики (tokio, `sd_listen_fds`, обработка сигналов) сломало бы эту
лёгкую, публикуемую идентичность.

Разделение:

- **`arona::lifecycle` (протокольный контракт, живёт в arona).** Только
  JSON-RPC-методы и типы: `DrainState`, `ReadyStatus`, `Lifecycle.Drain`,
  `Lifecycle.Status`, `Worker.Status` и т.д. Удовлетворяет правилу arona
  «спарено с обеих сторон».
- **`malkuth` (новый крейт, runtime).** Зависит от протокольных
  типов `arona` + `tokio` + привязки `libsystemd` (socket activation) +
  trait-ы бэкендов. С feature-флагами:
  - `replica` — координация и оркестрация подсистемы A.
  - `leader-follower` — выборы по lease и переключение при отказе подсистемы B.
  - `socket-activation` — получение fd от systemd.
  - `file-lock` / `pg-lock` / `lease` — бэкенды `CoordinationLock`.

Все три проекта зависят от `malkuth` и включают нужные им features
(см. матрицу в §8). Помещение всего в arona заставило бы его стать
«протокол + опциональный runtime» и разрушило бы его чистоту — не
рекомендуется.

## 5. Ключевые абстракции

### 5.1 `Worker` — супервизируемый ресурс-дочерний процесс

`Worker` — это один независимо завершаемый процесс, который удерживает
ровно один ресурс (соединение с PLC, последовательный порт, локальный
слушающий порт, sidecar вроде cosmos / pglite-proxy). Процесс — это
**граница изоляции сбоев**: баг в стеке Modbus не может отравить воркер
S7comm.

FSM жизненного цикла (взят из `evernight/src/model_server.rs:128-139`):

```
        запуск                     health ok
 Starting ──────► Running ─────────────────► Running
     │              │  ▲                          
     │              │  │ health ok (самовосстановление)    
     │              ▼  │                          
     └──────► Failed ◄┘        крах / нездоров
                  │                              
                  │ политика перезапуска = permanent    
                  └────────► Starting (rate-limited)
```

### 5.2 `Supervisor` — владеет пулом воркеров

- **Политика перезапуска** (терминология OTP): `permanent` (всегда
  перезапускать — по умолчанию для ресурсных воркеров), `transient`
  (перезапускать только при аварийном выходе), `temporary` (никогда не
  перезапускать).
- **Rate-limiting скользящим окном** (взято из entelecheia `health_daemon`
  `max_restart_attempts` + `cooldown`): если воркер перезапускается более
  N раз за окно W, он переходит в `cooldown`, чтобы предотвратить шторм
  крашей; последующие перезапуски откладываются.

### 5.3 `Lifecycle` — единая семантика сигналов (Layer 1)

Примем соглашение nginx/Go:

| Сигнал | Семантика | Поведение |
|---|---|---|
| `SIGINT` (ctrl_c) | эквивалент SIGTERM (удобно для разработки) | войти в дрейн |
| `SIGTERM` | **мягкое завершение** | дрейн: сбросить ready-бит → прекратить accept → дрейнить находящиеся в работе → выйти |
| `SIGHUP` | **горячая перезагрузка конфигурации** | не выходить; перечитать конфигурацию |
| `SIGQUIT` | **немедленный выход** (только в экстренных случаях) | пропустить дрейн, быстрый выход |

**Последовательность дрейна** (одна из реализаций; каждый проект
внедряет своё «замыкание дрейна»):

1. Установить `/readyz` `draining = true` (LB / оркестратор видит это и
   перестаёт слать новый трафик).
2. Прекратить `accept` новых соединений (при socket activation:
   прекратить accept из унаследованного fd).
3. Отправить close-кадры живым WebSocket-ам; ждать выполнения находящихся
   в работе запросов с таймаутом `DRAIN_TIMEOUT` (по умолчанию 30с,
   настраивается).
4. Дрейнить фоновые задачи (взять из entelecheia `TaskManager.stop_all`
   + `wait_all`).
5. Чисто отключить пулы upstream-соединений (взять чистое отключение из
   shittim-chest `upstream_pool`).
6. Освободить замки, очистить временные файлы → выйти с кодом 0.

Замечание по реализации: axum `axum::serve(listener,
app).with_graceful_shutdown(...)` уже поддерживает дрейн; **ключевой
недостающий элемент — подключить `SIGTERM`** (сегодня подключён только
`ctrl_c`). Ссылки: `entelecheia/.../shutdown.rs:17`,
`shittim-chest/.../api.rs:465`, `evernight/src/api/mod.rs:109`.

### 5.4 Эндпоинты здоровья (унифицированные)

Разделение проб (сегодня у трёх проектов оно несовместимо):

| Эндпоинт | Семантика | Решение |
|---|---|---|
| `/healthz` (liveness) | процесс жив | 200, если процесс может ответить (простой критерий перезапуска) |
| `/readyz` (readiness) | **может обслуживать**, несёт бит дрейна | 200, если не дрейнит И зависимости готовы (DB ping / сокет scepter / первый опрос станции); 503 во время дрейна |

Бит `draining` в `/readyz` — центральный сигнал скользящего обновления:
оркестратор направляет новые запросы только на экземпляры, у которых
`/readyz` равен 200. Существующий `GET /api/health` в shittim-chest
(`routes.rs:27`) модернизируется до `/readyz` с битом дрейна.

### 5.5 `acquire_listener` — бесшовная передача в Layer 3

`malkuth` предоставляет `acquire_listener(addr) -> TcpListener`:

1. Попытаться `sd_listen_fds()` (проверить `LISTEN_PID`) — systemd держит
   fd.
2. Откатиться к обычному `TcpListener::bind(addr)` (dev, без systemd).

axum `serve(listener, ...)` уже принимает пред-привязанный listener,
поэтому вся инфраструктура есть; сегодня не хватает только *источника*
fd. Три адаптера развёртывания:

| Развёртывание | Подход | Применимо к |
|---|---|---|
| **голый systemd** | `xxx.socket` + темплат-инстансы `xxx@.service` | scepter, evernight-gateway, сам malkuth |
| **docker** (shittim-chest prod) | socket activation на хосте от systemd, передача привязанного сокета/fd в контейнер (`LISTEN_FDS` + `SocketUser`); либо лёгкий in-container master, держащий fd | shittim-chest prod |
| **dev** | откат к обычному `bind` + короткое перекрытие (допустить несколько сотен мс потерянных соединений), без systemd | dev всех трёх проектов |

Скользящее обновление при socket activation:

```
[триггер обновления] → запуск нового экземпляра (темплат-сервис service@new, наследует/переполучает fd)
                     → опрос /readyz нового экземпляра до 200
                     → SIGTERM старому экземпляру (= дрейн)
                     → старый экземпляр дрейнит и самоуничтожается
                     → systemd держит fd всё время → ноль потерянных соединений
```

### 5.6 `CoordinationLock` — trait уровня L2 с бэкендами

Во время окна скользящего обновления старые и новые экземпляры могут
параллельно читать/писать общие ресурсы. Транзакции БД естественно
безопасны; файлам (evernight JSONL, конфиги) нужно «уведомление перед
записью + замок».

```rust
pub trait CoordinationLock: Send + Sync {
    async fn acquire(&self, key: &str, lease: Duration) -> Result<LockGuard>;
}
// Бэкенды:
//   FileLock  — flock/fcntl,       для evernight (JSONL / конфиг)
//   PgLock    — pg_advisory_lock,  для entelecheia / shittim-chest
//   LeaseLock — файловый замок + lease (авто-истечение при крахе)
```

**Реестр экземпляров** (используется только во время окна обновления; в
устойчивом состоянии — одна запись): небольшая общая таблица/файл,
записывающая `{instance_id, role: Active | Draining, started_at,
generation}`. Новый экземпляр записывает строку `Active` при старте; при
обновлении старая помечается `Draining`. Это заменяет кворум Raft, который
был явно вынесен за рамки, — поскольку в устойчивом состоянии экземпляр
один, реестр лишь координирует дрейн, без потребности в строгой
согласованности (достаточно файла или строки БД).

## 6. Подсистема A — Replica (балансировка нагрузки ⊃ скользящее обновление)

**Форма.** N идентичных равноправных экземпляров, работающих параллельно за
фронтальным LB, состояние в общем Postgres. **active-active**, узлы
равноправны, без лидера.

| Аспект | Подход |
|---|---|
| Маршрутизация запросов | фронтальный LB (caddy / встроенный round-robin `SO_REUSEPORT`); снятие по `/readyz` |
| Чтение/запись общего состояния | транзакции БД + `pg_advisory_lock` для координации параллельных записей (естественно безопасно) |
| WebSocket / длинные соединения | **sticky session** (LB фиксирует по cookie/instance id) или **миграция сессии** (после разрыва клиент переподключается к любой реплике; состояние восстанавливается из БД — `upstream_pool` в shittim-chest уже является шаблоном переподключения) |
| Сессионная близость | entelecheia и shittim-chest обе выносят состояние в Postgres и восстанавливают его при загрузке → **естественно дружественны к репликам**, их ключевое преимущество над evernight |
| Скользящее обновление | подоперация масштабирования реплик: добавить новую реплику (новой версии) → ready → дрейним и удаляем старую реплику → повторить. Миниатюра K8s `maxSurge/maxUnavailable` |

**Почему A сравнительно прост.** Поскольку entelecheia и shittim-chest
выносят состояние в Postgres (восстановление при загрузке, подтверждено
аудитом), репликам не нужна репликация состояния между ними — достаточно
фронтального LB плюс транзакций БД. Единственная сложная часть —
sticky/миграция WebSocket.

## 7. Подсистема B — Leader/Follower (периферийная active-passive HA)

**Форма.** Два процесса evernight на одном устройстве/шлюзе, один лидер,
один ведомый; upstream-сервер `evernight-server` видит **один `node_id`**
(одно устройство). **active-passive**, экземпляры неравноправны, лидер
эксклюзивно владеет физическим вводом-выводом.

### 7.1 «Трюк», упрощающий B: дерево супервизии + let-it-crash

Вместо обеспечения отказоустойчивости каждого ресурса, **только супервизор
делается leader/follower**; ресурсы — это независимые процессы-воркеры,
которые просто перезапускаются при крахе. Конкретно (согласно принятому
решению):

```
supervisor  (leader / follower HA)        ← только этот слой делает выборы по lease + переключение при отказе
   ├─ worker: PLC-A (Modbus)              ← супервизируемый перезапуск, один экземпляр
   ├─ worker: PLC-B (S7comm)              ← супервизируемый перезапуск, один экземпляр
   ├─ worker: serial / CAN                ← супервизируемый перезапуск, один экземпляр
   └─ worker: local port listener         ← супервизируемый перезапуск, один экземпляр
```

Это модель дерева супервизии OTP / «let it crash» (также systemd
`Restart=always`, K8s Pod). Преимущества:

- **Разделение ответственности.** Воркеры только «удерживают один ресурс и
  выполняют работу»; они не несут логики отказоустойчивости, выборов или
  синхронизации состояния — остаются максимально простыми.
- **Концентрация отказоустойчивости.** Только супервизор обеспечивает HA;
  сложность сводится в одно место.
- **Изоляция сбоев.** Крах одного ресурса (например, одного протокола PLC)
  не может затронуть другие (отдельные процессы).
- **Соответствие протоколам.** evernight говорит на множестве
  индустриальных протоколов (Modbus/S7/CAN/serial); отображение каждого на
  отдельный воркер даёт максимальную ценность изоляции.

### 7.2 Жизненный цикл воркера при переключении — модель дочернего процесса (согласованная отправная точка)

Воркеры — это **дочерние процессы** супервизора (`kill_on_drop`). Падение
лидера → воркеры остаются сиротами/убиваются → повышенный ведомый
**заново порождает всех воркеров** (каждый PLC переподключается).

- Простейшая модель; соответствует намерению «сервер ответственен».
- Цена: при переключении супервизора все ресурсы кратко прерываются и
  переподключаются.
- Продвинутое (отложено): воркеры как независимые демоны под более
  низкоуровневым init, супервизор лишь направляет их через IPC; новый
  супервизор `attach`-ит выживших воркеров. Меньше перебоев, но воркеры
  должны реализовать «перепривязку к текущему супервизору» — сложнее.
  Задокументировано как будущая опция.

### 7.3 Выборы лидера + fencing

- **Выборы по lease.** Лидер держит файловый замок + lease (с TTL),
  продлевает его при каждом heartbeat; ведомый опрашивает; по таймауту
  heartbeat лидера ведомый захватывает lease и повышает себя.
- **Эксклюзивный физический ресурс.** Соединения PLC/serial/CAN может
  удерживать только лидер (два процесса, опрашивающие один и тот же PLC,
  конфликтуют) → лидер опрашивает, ведомый в режиме ожидания. Это
  фундаментальная причина, по которой B — active-passive, а не
  active-active.
- **Синхронизация состояния.** В качестве отправной точки — **холодный
  резерв** (ведомый не реплицирует; при повышении восстанавливается из
  JSONL на диске). Горячий резерв (ведомый читает хвост JSONL лидера) —
  продвинутая опция.
- **Fencing при split-brain.** Lease TTL + fencing: ведомый может
  захватить лидерство только после истинного истечения lease; после
  захвата старый лидер физически лишается возможности дальнейшей записи
  (эксклюзивность физического ввода-вывода — естественный fence).
- **Единая идентичность устройства.** Лидер и ведомый делят один `node_id`;
  только текущий лидер отправляет `device.register`.

Классические аналоги: keepalived/VRRP, DRBD+Pacemaker, MySQL
primary/replica, Redis Sentinel — как внутри-машинная, процессно-уровневая
упрощённая версия. Теория (выборы по lease + fencing) хорошо обоснована.

| shittim-chest (chest) | одна из реплик | pglite-proxy (mock), channel intake | **A Replica** |

| evernight-server (центральный) | одна из реплик | контейнеры model_server | **A Replica** |

Выбор features по проектам (`malkuth`):

- entelecheia / shittim-chest / evernight-server: `replica` +
  `socket-activation` + `pg-lock`; абстракция Worker для их sidecar-ов.
- устройство evernight: `leader-follower` + `socket-activation` (продвинутый

## 9. Этапы внедрения

1. **Этап A — Layer 1 во всех трёх проектах.** Семантика сигналов +
   `/healthz` / `/readyz` + дрейн. Наименьший риск, наивысший немедленный
   эффект (сначала чиним жёсткое убийство по SIGTERM).
2. **Этап B — протокол `arona::lifecycle` + каркас `malkuth`.**
   Определения trait-ов, `acquire_listener`, trait `CoordinationLock` +
   бэкенды `FileLock` / `PgLock`, примитивы `Worker` + `Supervisor`.
3. **Этап C — Layer 3.** Юниты socket activation для трёх проектов +
   docker-адаптер + dev-откат.
4. **Этап D — Layer 4.** Оркестратор очереди манифестов (взять из
   `health_daemon`) + цикл дрейна сосуществования старого/нового, включая
   dev-процесс «скомпилировать новый сервер»; плюс переключение
   leader/follower для B.

## 10. Риски и границы

- **shittim-chest docker + socket activation** — самая неопределённая
  часть; передача fd в контейнер требует прототипа-spike. Если невозможно,
  откатиться к «внешний caddy + снятие по `/readyz` + короткое перекрытие».
- **In-memory состояние evernight** (`DeviceRegistry`, сессии) теряется при
  дрейне; нужно решить, сохранять или мигрировать (худший случай: новый
  экземпляр перестраивается, длинные сессии рвутся).
- **Дрейн WebSocket ≠ миграция.** Находящиеся в работе WS всё равно рвутся
  при выходе старого экземпляра; «бесшовность» требует, чтобы клиент
  переподключился к новому экземпляру (клиент shittim-chest уже имеет логику
  переподключения `upstream_pool`, переиспользуемо).
- **Процессно-уровневая vs потоко-уровневая отказоустойчивость.** Leader/follower
  (подсистема B) решает задачу *процессно/инстансно-уровневой*
  отказоустойчивости, а не потоко-уровневой. Panic в задаче tokio — это
  работа супервизора (перезапуск задачи, а не переключение процесса). Не
  используйте B для поглощения крашей потоков — это слишком тяжело.
- **Явно за рамками.** Кворум Raft, шардирование на consistent-hash,
  кросс-датацентровая HA — исключены областью «скользящее обновление +
  периферийная HA». Одноинстансное устойчивое состояние означает, что
  реестр лишь координирует дрейн.

## 11. Открытые вопросы (отложенные)

- Воркер при переключении: для отправной точки выбрана модель дочернего
  процесса; модель «независимый демон + attach» задокументирована как
  продвинутая.
- Холодный vs горячий резерв для B: для отправной точки выбран холодный
  (восстановление из JSONL при повышении); горячий (ведомый читает хвост
  лога лидера) отложен.
- Оборачивать ли sidecar `cosmos` из entelecheia и `pglite-proxy` из
  shittim-chest под единую абстракцию `Worker`: согласовано — да
  (объединить в единую абстракцию воркера во всех трёх проектах).
